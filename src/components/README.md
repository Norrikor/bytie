# Компоненты frontend

Единая точка входа: **`src/components/`**. Роуты (`src/app/...`) только склеивают страницы и данные; переиспользуемый UI живёт здесь.

## Структура

| Папка | Назначение |
|-------|------------|
| **`ui/`** | Приложение-агностичные виджеты: оверлеи, поля дат/времени, подсказки, простые паттерны навигации (`PageBackLink`). Без доменной модели «питомцы / события». |
| **`app/`** | Экранные блоки авторизованного приложения: карточки действий, лента, нижняя навигация, диалоги удаления в контексте объектов и т.д. Допускаются серверные и клиентские компоненты рядом. |

Импорт всегда через алиас проекта:

```ts
import Tooltip from '@/components/ui/Tooltip'
import ActionEventCard from '@/components/app/ActionEventCard'
```

Не добавлять новые **`_components`** рядом с роутами — это размазывало дерево и дублировало «где искать кнопку».

## Куда класть новый компонент

1. **Нужен в нескольких фичах и не знает про Prisma / «объект ухода»** → `ui/`.
2. **Про продукт (лента, объект, онбординг объекта)** → `app/`.
3. Сомневаешься — начни с `app/`; при обобщении вынеси в `ui/` без доменных имён в пропсах.

## Соглашения

- Один основной экспорт на файл по умолчанию (`export default`), как в текущих файлах.
- Стили — глобальные классы из `globals.scss` и локальные `*.module.scss` только если появится явный модульный стиль (сейчас преимущественно globals).
- Клиентские границы: `'use client'` только в файлах, где нужны хуки или браузерные API; серверные обёртки (данные из БД) остаются в `app/` страницах или в серверных компонентах в `components/app/` (например `BottomNav`).

Обновляй этот файл, если меняется правило — так агенты и люди держат одну картину.
